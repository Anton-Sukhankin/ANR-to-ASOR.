# Спецификация задачи: создание АСОР по АНР из Контур.X

Источник: транскрипция встречи "справка об ошибках из АНР в АСОР" от 17 июня 2026 г.

## 1. Краткое описание

Нужно спроектировать процесс, в котором пользователь инициирует задачу в Контур.X, прикладывает задание на АСОР и Excel-файл АНР, а в Cost/ESCost автоматически создается АСОР со строками, рассчитанными на основании АНР.

Решение должно не только создать строки сметы, но и объяснить пользователю, что именно было создано автоматически, какие решения система приняла уверенно, где есть предупреждения, ошибки или неоднозначные сопоставления, и какие места требуют проверки сметчиком.

## 2. Цель продукта

Сократить ручную работу сметчика при создании АСОР по АНР, сохранив контроль человека там, где исходные данные неполные, неоднозначные или не совпадают со справочниками.

Для ближайшего демо цель уже: показать полный путь от прикладывания документов в Контур.X до созданного АСОР в Cost/ESCost, используя подготовленные идеальные или почти идеальные данные.

## 3. Пользователи и участники

- Инициатор задачи в Контур.X: прикладывает документы и запускает процесс.
- Сметчик: проверяет созданный АСОР, разбирает предупреждения, подтверждает или исправляет спорные строки.
- Аналитик/демонстратор: показывает полный путь решения Олегу/бизнес-заказчику.
- Система Cost/ESCost: принимает структурированные данные, создает строки АСОР, ведет журнал обработки.
- Парсер АНР: преобразует Excel-файл АНР в JSON, пригодный для дальнейшей обработки.
- ИИ/автоматизированная логика: анализирует АНР, сопоставляет строки со справочниками, создает строки и формирует объяснения решений.

## 4. Границы первой версии для демо

### Входит в первую версию

- Запуск процесса из Контур.X на стейджовой среде.
- Прикладывание задания на АСОР и Excel-файла АНР.
- Передача данных в Cost/ESCost.
- Парсинг АНР в JSON.
- Создание АСОР на основании полученного JSON.
- Создание строк АСОР по данным АНР.
- Заполнение объемов и цен по найденным соответствиям.
- Отображение признака, что АСОР или отдельные строки сгенерированы автоматически.
- Лог обработки: что было получено, какие строки созданы, какие решения приняты, где возникли предупреждения.
- Подготовка демо на контролируемых данных.
- Резервная запись демо на случай нестабильности интеграций.

### Не входит в ближайшую первую версию

- Полная реализация проверки комплектности документов.
- Полный жизненный цикл АНР как самостоятельного документа в Cost.
- Автоматическое исправление всех некачественных данных.
- Исключение сметчика из процесса при наличии предупреждений или сомнений.
- Полное покрытие всех специализаций и всех форматов АНР.
- Генерация сущностей, которых нет в справочниках и которые система не может надежно сопоставить.

## 5. Входные данные

Обязательные входные данные:

- Excel-файл АНР.
- Задание на АСОР.
- Контекст задачи из Контур.X.
- Данные договора, объекта, подрядчика, специализации.

Дополнительные источники для сопоставления:

- Справочники групп работ.
- Справочники видов работ.
- Справочники материалов.
- Мэппинги между АНР и внутренними сущностями Cost/ESCost.
- Предыдущие или существующие строки АСОР по тому же контексту, если они доступны.

## 6. Выходные данные

Основной результат:

- Созданный АСОР в Cost/ESCost.

Внутри АСОР должны быть доступны:

- строки, созданные автоматически;
- заполненные объемы;
- заполненные цены;
- признак автоматической генерации;
- список предупреждений;
- список ошибок;
- журнал обработки;
- ссылка или представление исходного АНР;
- при возможности - промежуточная структурированная версия АНР.

## 7. Основной пользовательский путь

1. Пользователь открывает задачу в Контур.X.
2. Пользователь прикладывает задание на АСОР.
3. Пользователь прикладывает Excel-файл АНР.
4. Пользователь отправляет задачу в Cost/ESCost.
5. Система получает документы и запускает обработку.
6. Парсер преобразует АНР в JSON.
7. Cost/ESCost создает АСОР.
8. Система создает строки на основании АНР.
9. Система сопоставляет группы работ, виды работ, материалы, объемы и цены.
10. Система формирует журнал решений, предупреждений и ошибок.
11. Пользователь открывает созданный АСОР.
12. Пользователь видит плашку "Сгенерировано автоматически" или аналогичный статус.
13. Пользователь просматривает созданные строки и результаты сопоставления.
14. Если предупреждений нет, АСОР может идти дальше по процессу с минимальной ручной проверкой.
15. Если есть предупреждения или ошибки, сметчик проверяет их, подтверждает или исправляет строки.

## 8. Логика обработки

### 8.1. Преобразование АНР в JSON

Парсер должен преобразовать Excel-файл АНР в плоскую структуру данных.

Ожидаемая структура:

- метаданные АНР: договор, объект, подрядчик, специализация и другие атрибуты;
- список строк;
- для каждой строки: группа работ, вид работ, материал, объем, единица измерения, цена, признаки типа строки и исходные значения из Excel.

Плоская структура предпочтительна, потому что в исходном АНР группа работ может отсутствовать или быть неоднозначной.

### 8.2. Создание строк АСОР

На основании JSON система создает строки АСОР. Для каждой строки нужно:

- определить тип строки;
- найти соответствующую группу работ;
- найти соответствующий вид работ;
- найти соответствующий материал, если он требуется;
- определить объем;
- определить цену;
- сохранить связь с исходной строкой АНР;
- записать результат обработки в лог.

### 8.3. Общий порядок сопоставления

Для групп работ, видов работ и материалов базовый порядок должен быть таким:

1. Сначала искать совпадение внутри уже существующих или предыдущих строк АСОР по тому же контексту.
2. Затем искать точное совпадение в справочниках и мэппингах.
3. Если точного совпадения нет, использовать полнотекстовый поиск или поиск по похожести.
4. Если совпадение не 1 в 1, создавать предупреждение.
5. Если подходящего совпадения нет, не генерировать сущность из воздуха, а передавать строку на ручной разбор.

### 8.4. Группы работ

Группа работ - самый рискованный участок логики, потому что ее может не быть в АНР, но для создания строки АСОР ее нужно определить.

Правила:

- Если группа работ найдена точно, использовать ее без предупреждения.
- Если группа работ отсутствует или неоднозначна, сначала искать аналогичную группу в предыдущем/существующем АСОР.
- Если в АНР есть другие виды работ с однозначно определенной группой, использовать их как контекст.
- "Соседние виды работ" в этой логике означают все виды работ внутри данного АНР, а не только строки рядом физически.
- Если для проблемного вида работ найден список возможных групп, а среди других видов работ есть единственная группа, которая входит в этот список, выбрать ее.
- Если найдено несколько равноправных групп, создать предупреждение и показать пользователю альтернативы.
- Если система вынужденно выбирает группу не по точному совпадению, это всегда предупреждение.

Открытый вопрос для согласования: можно ли в неоднозначном случае автоматически выбирать группу с максимальной оценкой похожести или первый вариант, если сметчик все равно сможет исправить решение.

### 8.5. Виды работ

Правила:

- Вид работ должен сопоставляться со справочником или мэппингом.
- При точном совпадении предупреждение не нужно.
- При неточном совпадении допустим полнотекстовый поиск с оценкой уверенности.
- Если найдено несколько вариантов, выбирать вариант с максимальной оценкой уверенности и создавать предупреждение.
- Если вид работ не найден в справочниках, система не должна придумывать новый вид работ за сметчика.

### 8.6. Материалы

Правила:

- Материал сопоставляется по справочнику, мэппингу и контексту вида работ/группы работ.
- Если найдено несколько вариантов, выбирать вариант с максимальной оценкой уверенности и создавать предупреждение.
- Если материал не найден или не соответствует справочнику, система не должна автоматически заменять его на "правильный" без проверки.
- Материал не всегда обязан быть назначен автоматически, в отличие от группы работ.

### 8.7. Числовые значения

Система должна уметь нормализовать распространенные форматы чисел:

- десятичная запятая;
- десятичная точка;
- разделители тысяч;
- пробелы и лишние символы вокруг чисел.

Ошибки формата числа не должны ломать весь процесс, если значение можно надежно распознать.

## 9. Предупреждения, ошибки и уверенность

### Предупреждение

Предупреждение появляется, когда система смогла принять решение, но оно не является полностью надежным.

Примеры:

- совпадение найдено не 1 в 1;
- выбран вариант с максимальной оценкой похожести;
- были альтернативные варианты;
- группа работ определена по контексту других строк;
- справочник дал низкую или среднюю уверенность;
- значение было нормализовано из нестандартного формата.

Если есть хотя бы одно предупреждение, сметчик остается в процессе и должен подтвердить или исправить результат.

### Ошибка

Ошибка появляется, когда система не может создать корректную строку без участия человека.

Примеры:

- обязательная группа работ не определена;
- вид работ не найден и нет надежных кандидатов;
- критичные данные отсутствуют;
- строка не может быть преобразована в допустимый формат АСОР;
- исходный файл не читается.

### Уровень уверенности

Уровень уверенности нужен для объяснения решений в интерфейсе и для настройки строгости.

Варианты использования:

- показывать уверенность в карточке предупреждения;
- настраивать порог, ниже которого появляется предупреждение;
- разрешать более строгий режим, где любое неточное совпадение становится предупреждением.

Для первой версии можно считать, что любое решение не по точному совпадению создает предупреждение.

## 10. Задача пользовательского опыта

Интерфейс должен помогать пользователю быстро понять:

- что было создано автоматически;
- на основании каких исходных данных это создано;
- где система уверена;
- где система сомневается;
- что именно должен проверить сметчик;
- какие действия доступны для исправления.

Ключевой принцип: автоматизация не должна быть "черным ящиком". Пользователь должен видеть объяснение решения и иметь возможность вмешаться.

## 11. Основные экраны и элементы интерфейса

### 11.1. Карточка/страница созданного АСОР

Элементы:

- статус создания;
- плашка "Сгенерировано автоматически";
- источник создания: Контур.X, задание на АСОР, Excel АНР;
- краткая сводка: сколько строк создано, сколько предупреждений, сколько ошибок;
- ссылка на журнал обработки;
- ссылка на исходный АНР;
- ссылка на промежуточную структурированную версию, если она создается.

### 11.2. Журнал обработки

Журнал должен быть понятен не только разработчику, но и аналитику/сметчику.

Элементы:

- временная шкала обработки;
- шаг "Получен Excel АНР";
- шаг "АНР преобразован в JSON";
- шаг "Создан АСОР";
- шаг "Созданы строки";
- шаг "Заполнены цены и объемы";
- шаги с предупреждениями и ошибками;
- возможность открыть связанную строку АСОР или исходную строку АНР.

### 11.3. Таблица строк АСОР

Элементы:

- признак автоматически созданной строки;
- группа работ;
- вид работ;
- материал;
- объем;
- цена;
- источник значения;
- статус строки: без замечаний, предупреждение, ошибка;
- фильтры по статусам;
- фильтр "только сгенерированные";
- переход к деталям сопоставления.

### 11.4. Карточка предупреждения

Карточка должна отвечать на вопросы:

- что система пыталась определить;
- какое значение было в исходном АНР;
- какое значение выбрано в АСОР;
- почему выбрано именно оно;
- какая уверенность у решения;
- какие были альтернативы;
- что нужно сделать пользователю.

Действия:

- подтвердить выбор;
- выбрать другой вариант;
- открыть исходную строку АНР;
- открыть строку АСОР;
- оставить комментарий;
- отправить на ручной разбор.

### 11.5. Сравнение АНР и АСОР

Желательный элемент интерфейса для доверия к результату:

- слева исходная строка АНР;
- справа созданная строка АСОР;
- подсветка автоматически заполненных полей;
- подсветка полей с предупреждениями;
- объяснение источника цены и объема.

## 12. Пользовательские сценарии

### Сценарий 1. Демо на идеальных данных

1. Демонстратор открывает Контур.X stage.
2. Прикладывает задание на АСОР и подготовленный Excel АНР.
3. Отправляет задачу в Cost/ESCost.
4. Открывает созданный АСОР.
5. Показывает плашку автоматической генерации.
6. Показывает созданные строки, цены и объемы.
7. Показывает журнал обработки.
8. Объясняет, что на идеальных данных система может пройти путь почти без участия сметчика.

### Сценарий 2. Сметчик проверяет предупреждения

1. Сметчик открывает созданный АСОР.
2. Видит счетчик предупреждений.
3. Открывает список предупреждений.
4. Проверяет карточку предупреждения.
5. Сравнивает исходную строку АНР и созданную строку АСОР.
6. Подтверждает выбор системы или исправляет его.
7. После обработки всех предупреждений переводит АСОР дальше по процессу.

### Сценарий 3. Система не смогла сопоставить обязательные данные

1. Система создает АСОР частично или останавливает создание проблемной строки.
2. В журнале появляется ошибка.
3. Пользователь открывает ошибку.
4. Интерфейс показывает, какое обязательное поле не определено.
5. Пользователь вручную выбирает значение или отправляет строку на разбор.

### Сценарий 4. Настройка строгости сопоставления

1. Аналитик или администратор задает порог уверенности для полнотекстового поиска.
2. Система применяет порог к группам работ, видам работ и материалам.
3. При снижении уверенности ниже порога создается предупреждение.
4. При строгом режиме любое неточное совпадение становится предупреждением.

## 13. Состояния интерфейса

- Документы получены, обработка ожидает запуска.
- Идет парсинг АНР.
- АНР успешно преобразован в JSON.
- Создается АСОР.
- АСОР создан без предупреждений.
- АСОР создан с предупреждениями.
- АСОР создан с ошибками.
- Обработка завершилась ошибкой.
- Требуется ручная проверка сметчика.
- Все предупреждения разобраны.

## 14. Требования к прозрачности и доверию

- Каждое автоматическое решение должно иметь источник.
- Для спорного решения должны быть видны альтернативы.
- Пользователь должен понимать, почему появилась ошибка или предупреждение.
- Нельзя скрыто заменять значения, которые не найдены в справочниках.
- Нельзя обещать полное исключение человека, если есть предупреждения или сомнения.
- Для демо нужно честно проговорить, что масштабирование зависит от качества АНР и доли Excel-файлов подходящего формата.

## 15. Критерии приемки первой версии

- Из Контур.X можно инициировать задачу с заданием на АСОР и Excel АНР.
- Cost/ESCost получает данные и создает АСОР.
- АНР преобразуется в JSON.
- По JSON создаются строки АСОР.
- Для созданных строк заполнены ключевые поля: группа работ, вид работ, материал при наличии, объем, цена.
- Созданные строки визуально отличаются от ручных или имеют явный признак генерации.
- Пользователь видит журнал обработки.
- Пользователь видит предупреждения и ошибки.
- Предупреждение содержит объяснение и действие для пользователя.
- Ошибка не маскируется под успешную обработку.
- Есть подготовленный сценарий демо и резервная запись.

## 16. Открытые вопросы

- Где именно должен отображаться журнал обработки: только в АСОР, в отдельной вкладке или в общем логе задачи?
- Нужно ли создавать промежуточный документ АНР в Cost уже в первой версии или достаточно JSON и лога?
- Какая финальная структура JSON нужна Никите как вход для создания строк?

## 17. Риски

- Реальные АНР могут быть не в Excel или иметь сильно разные структуры.
- Названия в АНР могут не совпадать со справочниками 1 в 1.
- Автоматическое сопоставление может создать неверную строку, если не показывать предупреждение.
- Полный путь через Контур.X и Cost/ESCost зависит от нескольких интеграций и может быть нестабилен на демо.
- Без понятного журнала обработки бизнес может не увидеть ценность автоматизации.
- Без управляемого пользовательского опыта предупреждений сметчик получит не помощь, а новый источник ручной проверки.

## 18. Рекомендуемая формулировка задачи для проектирования

Спроектировать интерфейс и логику обработки, которые позволяют автоматически создать АСОР по Excel АНР, приложенному в Контур.X, показать результат в Cost/ESCost, объяснить каждое автоматическое решение через журнал и карточки предупреждений, а также дать сметчику понятный путь подтверждения или исправления спорных сопоставлений.

Ключевой фокус пользовательского опыта: пользователь должен быстро отличать надежно созданные строки от строк, где система сомневается, и видеть, что именно нужно проверить.
